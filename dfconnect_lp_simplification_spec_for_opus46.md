# DF CONNECT LP短縮・問い合わせ導線最適化 仕様書 / Opus 4.6向け

- 対象サイト: `https://dfconnect.jp/`
- 対象ページ: DF CONNECT ホワイトラベル制作・運用保守代行LP
- 想定実装担当: Opus 4.6
- 作成日: 2026-05-10
- 目的: 現行LPを、問い合わせしやすい短いオプトインLP寄りの構成へ再設計する

---

## 0. このmdの使い方

このmdは、LP周りの改修をOpus 4.6へ依頼するための仕様書である。

Opus 4.6には、以下を守ってLP構成・コピー・HTML/CSS・フォームUIを修正させる。

- LPの情報量を大幅に削る
- 問い合わせ前の心理的ハードルを下げる
- フォーム項目を最小限にする
- Why / What / How / Form の流れで、人間が迷わず理解できるLPにする
- 「対応外です」「やりません」ではなく、制作会社側が安心できる言い方に変える
- PHP送信処理そのものは別仕様に合わせる。Opus 4.6はLP表示・フォームUI・フォームHTMLの整理を主担当とする

---

## 1. 改修の最重要方針

### 1.1 LPの役割

現行LPは、サービス説明・守秘方針・デモサイト・匿名実績・対応範囲・対応スタイル・料金・ワークフロー・技術スタック・対応外・セキュリティ・FAQ・フォームまで情報が多い。

新LPでは、LPの役割を以下に絞る。

```txt
Web制作会社・広告代理店の担当者が、
「案件概要だけ送って相談してみよう」と思える状態を作る。
```

LPの目的は、すべてを説明し切ることではない。
初回問い合わせに必要な不安だけを潰し、入力負担を最小化することが目的である。

### 1.2 新LPで伝えるべきこと

新LPでは、極論として以下だけ伝わればよい。

```txt
Why：なぜ今、外部制作パートナーが必要なのか
What：何を任せられるのか
How：どう安全に進めるのか
Form：概要だけ送れる導線
```

この4つに該当しない情報は、削除または統合する。

### 1.3 絶対に避けること

以下は禁止する。

- 詳細情報をFAQやアコーディオンに詰め直すこと
- 現行LPの各セクションをほぼそのまま短く並べ直すこと
- サービスカタログのような長いLPに戻すこと
- 技術スタックや対応範囲を細かく羅列すること
- 「できないこと」を冷たく強調すること
- デモサイト・匿名実績・FAQを主導線にすること
- フォーム項目を増やすこと
- 問い合わせ前に分類・予算・時期・対応スタイルを細かく入力させること

## 2. 改修後のゴール

### 2.1 ユーザー視点のゴール

LPを見たWeb制作会社・広告代理店の担当者が、以下のように感じる状態を目指す。

```txt
あ、うちみたいな制作会社向けだ。
手が足りない時に裏方で動いてくれそう。
NDA前でも社名やURLを伏せて相談していいんだ。
小さな修正や保守だけでも相談してよさそう。
フォームも短いし、とりあえず概要だけ送ってみよう。
```

### 2.2 ビジネス視点のゴール

- 問い合わせフォームへの到達率を上げる
- フォーム入力完了率を上げる
- 問い合わせ前の離脱要因を減らす
- 小規模相談・スポット相談も拾えるようにする
- ホワイトラベル案件に必要な安心感は残す
- ただし、LP全体は短くする

### 2.3 成功条件

以下を満たせば成功とする。

- ファーストビューで「Web制作会社向け」「貴社名義」「制作・改修・保守パートナー」が伝わる
- ファーストビューまたはその直後にフォームがある
- 表示フォーム項目が6項目だけになっている
- 「案件概要だけで相談OK」が明確に伝わる
- NDA前は社名・URL・詳細を伏せてよいことが伝わる
- エンドクライアント対応の線引きが冷たくなく伝わる
- 大規模開発は対象外だが、Web制作に付随する実装は対応することが自然に伝わる
- フッターが最小限になっている
- 全体が長い営業資料ではなく、問い合わせ導線中心のLPになっている

## 3. 対象ユーザー

### 3.1 主ターゲット

```txt
Web制作会社・広告代理店・デザイン会社の担当者
```

### 3.2 想定ユーザーの状況

- 制作リソースが一時的に足りない
- 下層ページ制作や軽微修正が溜まっている
- デザイン支給後のコーディングだけ外注したい
- WordPress改修やフォーム調整だけ頼みたい
- NDA案件・ホワイトラベル案件を任せられる外注先を探している
- エンドクライアント情報を守れる相手か不安
- まずは小さく相談したい
- 予算や時期がまだ固まっていない

### 3.3 このユーザーが問い合わせ前に気にすること

問い合わせ前の人が知りたいのは、主に以下である。

```txt
1. ここは制作会社向けなのか
2. 何を任せられるのか
3. クライアント情報を伏せても相談できるのか
4. エンドクライアント対応で立場を守ってくれるのか
5. 小さな作業でも相談してよいのか
6. フォーム入力が面倒ではないか
```

LPでは、この不安を短く潰す。
詳細説明を増やすのではなく、問い合わせの心理的負担を減らす。

## 4. 改修後の推奨ページ構成

新LPは、以下の構成を基本とする。

```txt
01. Header
02. Hero + Form
03. Why
04. What
05. How
06. 安心して進めるための前提
07. Final CTA
08. Footer
```

ただし、PCでは `Hero + Form` を同一画面内の2カラムにできる場合は優先する。
スマホでは、Heroコピー → CTA → フォームの順で表示する。

## 5. Header仕様

### 5.1 方針

ヘッダーは極限までシンプルにする。
現行のように複数ナビを並べない。

### 5.2 推奨構成

```txt
左：DF CONNECT ロゴ
右：相談する
```

または、必要最小限で以下まで許容する。

```txt
左：DF CONNECT ロゴ
右：対応内容 / 進め方 / 相談する
```

### 5.3 禁止

以下のナビ項目は基本的に出さない。

- サービス
- 選ばれる理由
- デモサイト
- 匿名実績
- 制作の流れ
- FAQ
- 品質確認
- 対応範囲

理由は、ページを「読む導線」に戻してしまうため。
新LPでは「相談する」導線を主役にする。

## 6. Hero仕様

### 6.1 Heroの役割

Heroでは、以下を即座に伝える。

```txt
誰向けか：Web制作会社・広告代理店向け
何をするか：貴社名義で制作・改修・保守を支援
なぜ安心か：NDA前は概要だけで相談OK
何をしてほしいか：フォームから概要を送る
```

### 6.2 メインコピー

以下を第一候補とする。

```txt
Web制作会社の手が足りない時に。
貴社名義で動ける制作・改修パートナー。
```

別案として以下も許容する。

```txt
制作会社の裏側で動く、
貴社名義の制作・改修パートナー。
```

ただし、コピーを抽象化しすぎないこと。
「伴走」「共創」「DX」など、何をするか分かりにくい言葉は使わない。

### 6.3 サブコピー

以下を使用する。

```txt
デザイン支給後のコーディング、WordPress改修、軽微修正・保守まで。
NDA前は社名・URLを伏せた概要だけでご相談いただけます。
```

### 6.4 Hero内の補足チップ

4つまでに絞る。

```txt
NDA対応
貴社名義で納品
必要時のみMTG同席
小さな修正から相談OK
```

5つ以上に増やさない。
説明文も付けない。

### 6.5 Hero CTA

CTA文言は以下を第一候補とする。

```txt
概要だけ送って相談する
```

別案。

```txt
対応可否を相談する
案件概要だけ送る
```

「お問い合わせ」「送信する」「依頼する」よりも、心理的負担の少ない文言を使う。

---

## 7. Form仕様

### 7.1 フォームの位置

フォームはLPの主役として扱う。

PCでは可能であればHero右側に配置する。

```txt
左：Heroコピー・価値訴求
右：フォームカード
```

スマホでは以下の順番とする。

```txt
Heroコピー
補足チップ
CTAまたはフォーム見出し
フォーム
```

フォームをページ最下部だけに置くのは避ける。
問い合わせ導線を早く見せる。

### 7.2 フォーム見出し

```txt
案件概要だけでもご相談ください
```

### 7.3 フォーム補足文

```txt
NDA締結前は、クライアント名・URL・詳細情報を伏せた内容で問題ありません。
対応可否と進め方を整理して返信します。
```

### 7.4 表示するフォーム項目

表示する項目は以下のみ。

| No | ラベル | name属性 | 種別 | 必須 | 備考 |
|---:|---|---|---|---|---|
| 1 | 会社名・屋号 | `company` | text | 必須 | フリーランス・個人事業主も入力しやすい表記にする |
| 2 | お名前 | `name` | text | 必須 | 担当者名 |
| 3 | メールアドレス | `email` | email | 必須 | 返信先 |
| 4 | ご相談内容 | `inquiry_type` | select | 必須 | 選択肢は下記固定 |
| 5 | 案件概要・メッセージ | `message` | textarea | 任意 | NDA前提のため任意にする |
| 6 | プライバシーポリシー同意 | `privacy` | checkbox | 必須 | 値は `1` 想定 |

### 7.5 `inquiry_type` の選択肢

表示文言は以下。

```txt
選択してください
コーディング・LP実装
WordPress制作・改修
運用保守・軽微修正
フォーム・公開前チェック
NDA・ホワイトラベル相談
まだ決まっていない
```

推奨value。

```txt
coding_lp
wordpress
maintenance
form_check
white_label_nda
undecided
```

### 7.6 削除するフォーム項目

以下はフォーム上に表示しない。

```txt
会社区分
希望の対応スタイル
希望時期
予算感
NDA希望
希望の連絡方法
```

### 7.7 削除理由

| 削除項目 | 削除理由 |
|---|---|
| 会社区分 | 会社名・屋号や相談内容から把握できる。初回入力で分類させる必要がない |
| 希望の対応スタイル | 完全裏方・MTG同席・要件整理は初回返信で整理すればよい |
| 希望時期 | 急ぎの場合は任意メッセージに書ける。必須化すると入力負担になる |
| 予算感 | もっとも入力を止めやすい項目。初回相談後に聞けばよい |
| NDA希望 | LP上で「NDA前は概要だけでOK」と伝えればよい |
| 希望の連絡方法 | 初回返信はメールでよい。必要なら後でMTG調整する |

### 7.8 hidden項目

セキュリティ上必要なhidden項目は追加してよい。
ただし、ユーザーに見えるフォーム項目としては扱わない。

許可するhidden項目例。

```txt
csrf_token
honeypot
form_started_at
```

表示フォーム項目は、あくまで6項目のみとする。

### 7.9 メッセージ欄のプレースホルダー

```txt
例：デザイン支給のLPコーディングを相談したいです。
NDA前のため、クライアント名やURLは伏せています。
```

### 7.10 送信ボタン

第一候補。

```txt
概要だけ送って相談する
```

別案。

```txt
対応可否を相談する
```

「送信する」だけにはしない。
人間が押しやすい、相談の軽さが伝わる文言にする。

## 8. フォームUI仕様

### 8.1 基本方針

フォームは大きく、見やすく、入力しやすくする。
小さい入力欄は避ける。
フォームはLPの主役であり、付け足しに見せない。

### 8.2 推奨サイズ

| 要素 | 推奨 |
|---|---|
| フォームカード幅 | PC最大560〜640px |
| フォームカード内余白 | 32〜40px |
| input / select 高さ | 56px前後 |
| textarea 高さ | 140〜160px |
| ラベル文字サイズ | 15〜16px |
| 入力文字サイズ | 16px以上 |
| 項目間余白 | 18〜22px |
| 送信ボタン高さ | 58〜64px |
| チェックボックス | 20px以上 |

### 8.3 スマホでの注意

- 入力文字サイズは16px以上にする
- iOSで入力時にズームされないようにする
- タップ領域を広くする
- フォームカード左右の余白を詰めすぎない
- 送信ボタンは横幅100%で押しやすくする
- チェックボックスとラベルの間隔を十分に取る

### 8.4 エラー表示

- エラーは該当項目の直下に表示する
- 色だけで伝えない
- エラー文は短くする
- 未入力時にユーザーを責める文言にしない

例。

```txt
会社名・屋号を入力してください。
お名前を入力してください。
メールアドレスを入力してください。
ご相談内容を選択してください。
プライバシーポリシーへの同意が必要です。
```

## 9. Whyセクション仕様

### 9.1 役割

Whyでは、制作会社の現場で起きる「外注したくなる理由」を短く提示する。
悩みを大量に並べない。
3項目までに絞る。

### 9.2 見出し

```txt
制作会社の現場で、こんな作業が詰まりやすくありませんか？
```

### 9.3 本文

```txt
新規案件だけでなく、下層ページ制作、既存サイトの改修、公開後の更新作業まで。
社内リソースを圧迫しやすい実務を、必要な範囲だけ外部化できます。
```

### 9.4 カードまたは箇条書き

3項目のみ。

```txt
案件はあるのに、手を動かせる制作リソースが足りない
軽微修正や保守作業が社内に溜まっている
外注したいが、クライアント情報や品質管理が不安
```

### 9.5 Whyの結論文

```txt
DF CONNECTは、制作会社様の裏側で、必要な作業だけを引き受ける外部制作パートナーです。
```

## 10. Whatセクション仕様

### 10.1 役割

Whatでは「何を任せられるか」を4分類で伝える。
細かい技術スタックや作業一覧にしない。

### 10.2 見出し

```txt
任せられること
```

### 10.3 4分類

#### 1. コーディング・LP実装

```txt
Figma / XD / 画像カンプからのHTML/CSS実装、レスポンシブ対応。
LPや下層ページの実装もご相談いただけます。
```

#### 2. WordPress制作・改修

```txt
固定ページ追加、既存テーマ修正、フォーム調整、軽微なPHP対応。
既存サイトの改修やWordPress化にも対応します。
```

#### 3. 運用保守・軽微修正

```txt
テキスト修正、画像差し替え、WordPress更新補助、バックアップ確認など。
公開後の細かな作業を外部化できます。
```

#### 4. 公開前チェック・フォーム確認

```txt
表示崩れ、リンク切れ、フォーム送信、OGP/meta、公開作業補助。
納品前・公開前の確認作業も支援します。
```

### 10.4 価格目安について

料金表カードは作らない。
入れる場合は1行だけにする。

第一候補。

```txt
小さな修正・公開前チェックなど、試しやすい範囲からご相談いただけます。
```

価格を出す場合の許容表現。

```txt
小さな修正・公開前チェックは10,000円〜、下層ページ実装・WordPress改修は30,000円〜ご相談可能です。
```

ただし、価格を強調しすぎない。
今回のLPの主目的は、価格比較ではなく「まず相談してもらうこと」である。

## 11. Howセクション仕様

### 11.1 役割

Howでは、どう安全に進めるのかを3ステップで伝える。
現行の5ステップより短くする。

### 11.2 見出し

```txt
概要だけ送って、必要な範囲から進められます
```

### 11.3 3ステップ

#### 1. 概要だけ送る

```txt
NDA前は、クライアント名・URL・詳細情報を伏せた内容で問題ありません。
```

#### 2. 対応範囲を整理する

```txt
作業内容、納期、連絡経路、MTG同席の有無を確認します。
```

#### 3. 必要に応じてNDA後に制作開始

```txt
貴社名義・貴社窓口の方針に合わせて進行します。
```

### 11.4 注意

Howでチャットツール・タスク管理ツール・ファイル共有ツールの羅列はしない。
必要なら以下の1文に圧縮する。

```txt
連絡・共有方法は、貴社の既存フローに合わせて調整します。
```

## 12. 安心して進めるための前提

### 12.1 役割

現行の「対応外・要相談」「セキュリティ」「エンドクライアント対応」「大規模開発対象外」を、冷たくない表現で1セクションに統合する。

### 12.2 見出し

```txt
安心して進めるための前提
```

または、より柔らかくするなら以下。

```txt
制作会社様の立場を守るために
```

### 12.3 掲載する内容

3項目まで。

#### 1. NDA前の相談

```txt
NDA締結前は、社名・URLなどの機密情報を伏せた概要のみで問題ありません。
```

#### 2. エンドクライアント対応

```txt
エンドクライアントとの連絡・契約は、原則として発注元企業様を窓口にして進行します。
必要に応じて、貴社の制作パートナーとして打ち合わせ同席や技術説明をサポートします。
```

#### 3. 開発範囲

```txt
対応範囲は、Web制作に付随する実装・改修・保守が中心です。
会員機能、決済、予約システム、業務システムなど、大規模なシステム開発は原則対象外です。
```

### 12.4 禁止表現

以下のような表現は使わない。

```txt
エンドクライアントには絶対対応しません。
開発はやりません。
対象外です。
不可です。
丸投げ禁止です。
```

線引きは必要だが、拒絶感を出さない。
制作会社側の立場を守るための進行ルールとして表現する。

## 13. Final CTA仕様

### 13.1 役割

LPを読んだ後、もう一度フォームへ戻す。
ただし、長いCTAセクションにはしない。

### 13.2 コピー

```txt
まずは概要だけ送ってください。
対応可否と進め方を整理して返信します。
```

### 13.3 ボタン

```txt
概要だけ送って相談する
```

クリック時はフォーム位置へスクロールする。
フォームがHero右側にある場合も、アンカーでフォームへ移動させる。

## 14. Footer仕様

### 14.1 方針

フッターは最小限にする。
現行のサイトマップ型フッターは不要。

### 14.2 推奨構成

```txt
DF CONNECT
Web制作会社・広告代理店向けの制作・改修・運用保守パートナー

会社概要 / プライバシーポリシー / 特定商取引法に基づく表記

© DF CONNECT All Rights Reserved.
```

### 14.3 削除するフッター項目

以下は出さない。

- サービス一覧
- 品質確認
- 選ばれる理由
- 制作の流れ
- お問い合わせフォームへの重複リンク一覧
- デモサイトリンク群

## 16. FAQの扱い

### 16.1 基本方針

FAQは原則削除する。
FAQを残す場合も最大3件まで。

### 16.2 残す場合のFAQ

#### Q1. NDA前でも相談できますか？

```txt
はい。クライアント名・URL・詳細情報を伏せた概要のみで問題ありません。
```

#### Q2. エンドクライアントとのやり取りはどうなりますか？

```txt
原則として発注元企業様を窓口に進行します。必要に応じて、貴社の制作パートナーとしてMTG同席や技術説明も可能です。
```

#### Q3. 小さな修正だけでも相談できますか？

```txt
はい。軽微修正、フォーム確認、公開前チェックなど、小さな作業からご相談いただけます。
```

### 16.3 注意

FAQを増やしてLPを長くしない。
FAQで説明したくなった内容は、基本的に本文へ吸収するか削る。

## 17. デモサイト・匿名実績の扱い

### 17.1 基本方針

デモサイト・匿名実績は、今回の短縮LPでは主導線にしない。

理由。

```txt
問い合わせ前にデモサイトや匿名実績へ寄り道すると、フォーム送信導線が弱くなるため。
```

### 17.2 残す場合の最小表現

```txt
実績非公開案件が多いため、品質確認用のデモサイトをご用意しています。
必要な方には個別にご案内します。
```

または、Heroのチップ程度にする。

```txt
品質確認用デモあり
```

ただし、CTAボタンとして「デモを見る」を主CTAと並べない。
新LPの主CTAは「概要だけ送って相談する」ひとつに絞る。

## 20. レスポンシブ方針

### 20.1 PC

推奨レイアウト。

```txt
Hero左：コピー・チップ・CTA
Hero右：フォームカード
```

コンテンツ幅は広げすぎない。
LP全体を短く見せるため、余白を取りつつも縦に冗長にしない。

### 20.2 タブレット

- Heroは2カラムまたは1カラムへ自然に切り替える
- フォーム幅を確保する
- チップやカードが詰まりすぎないようにする

### 20.3 スマホ

- 1カラム固定
- Hero直後にフォームを置く
- input/select/textareaは横幅100%
- CTAは押しやすくする
- セクション間の余白を取りすぎて縦長化しない
- カードは多用しない

## 21. アクセシビリティ

以下を満たす。

- `label` と `input/select/textarea` を正しく紐づける
- 必須項目は視覚的にもテキストでも分かるようにする
- エラー表示はaria属性またはテキストで明確にする
- CTAやリンクはキーボード操作可能にする
- 文字サイズを小さくしすぎない
- コントラストを確保する
- フォームの入力欄はスマホでも押しやすくする
- 画像を使う場合は適切なaltを設定する

## 23. フォーム実装互換性

### 23.1 formタグ

PHP側の送信先が確定している場合は、それに合わせる。
未確定の場合は以下の仮置きとする。

```html
<form action="/contact/submit.php" method="post">
```

### 23.2 name属性

以下のname属性は変更しない。

```txt
company
name
email
inquiry_type
message
privacy
```

### 23.3 任意化の注意

`message` は任意に変更する。

PHP側バリデーションが既存仕様で `message` 必須になっている場合、LP改修後はPHP仕様も以下に合わせて変更する必要がある。

```txt
message：任意、最大5000文字、空欄許可
```

### 23.4 削除項目の扱い

以下の旧項目は、フォームHTMLから削除する。
DB・メール側に既存カラムがある場合は、NULLまたは空欄で扱う。

```txt
company_type
response_style
desired_timing
budget
nda_preference
contact_method
```

※ 実際の旧name属性が異なる場合は、該当する旧項目を同様に削除する。

## 27. 最終判断

今回のLP改修では、情報の網羅性よりも、問い合わせのしやすさを優先する。

現行LPの方向性である「Web制作会社向け」「貴社名義」「NDA対応」「制作・改修・保守」は残す。
一方で、詳細なサービス説明、デモ、匿名実績、料金表、技術スタック、長いFAQ、細かいフォーム項目は削る。

最終的に目指すページは以下。

```txt
あ、うち向けだ。
概要だけ送っていいんだ。
フォーム短いし、今送れる。
```

この状態を作ることを最優先にする。
