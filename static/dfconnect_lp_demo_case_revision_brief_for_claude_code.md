# DF CONNECT ホワイトラベルLP改修指示書

## Claude Code / Opus 4.5 用ブリーフ

このMarkdownは、既存のDF CONNECT LPを「Web制作会社向けのホワイトラベル制作・運用保守パートナー」として、より信頼されるLPに改修するための詳細指示書です。

想定する作業者は Claude Code / Opus 4.5。  
このファイルをそのままプロンプトとして渡して、既存の静的LPコードを改修することを前提にしています。

---

# 0. 重要な結論

今回のLPは、単なる「HP制作できます」ではなく、以下の勝ち筋に完全に寄せる。

```txt
Web制作会社・広告代理店・フリーランス制作者の裏側で動く、
NDA対応・ホワイトラベル対応の制作〜運用保守パートナー。
```

そのため、LP上で最も重要なのは「華やかな制作実績の羅列」ではない。  
むしろ、以下の3つを整理して見せること。

```txt
1. 実案件は守秘・ホワイトラベル前提なので、公開実績としては大きく出さない
2. 代わりに、品質確認用のデモサイトをしっかり見せる
3. 実案件の経験値は、匿名の対応実績として“何を担当したか”で見せる
```

LPの中心思想はこれ。

```txt
公開実績を見せびらかす会社ではなく、
発注元のブランドと案件情報を守る裏方パートナー。
```

# 1. 今回の改修で絶対に守る方針

## 1-1. 実績URLはLPに直接載せない

下記2件は実際の納品・協力実績として確認材料にはなるが、LP上では原則として直接掲載しない。

```txt
https://mscreate.jp/docs/2024070800023/
https://mscreate.jp/docs/2024070800016/
```

理由：

```txt
・デザイン品質が今回のLPの訴求に対して弱く見える可能性がある
・元デザイン踏襲、既存サイト移設、CMS対応など、見た目で評価されると不利
・M's Create様経由の協力案件であり、DF CONNECT単独の実績として誤解される可能性がある
・ホワイトラベル／協力案件を売るLPで、許諾が曖昧な公開実績を前面に出すと逆に不信感になる
・Web制作会社側から「うちの案件も表に出されるのでは？」と思われるリスクがある
```

したがって、LPでは実名・URL・スクリーンショットを掲載しない。  
使う場合でも、問い合わせ後の商談時に「匿名化した対応内容」として説明する程度に留める。

---

## 1-2. 実績は「匿名対応実績」として扱う

実案件は、見た目のデザインではなく、担当範囲・進行力・技術対応・裏方対応力を見せる。

表記例：

```txt
公開不可の協力案件として、既存サイトのCMS移行・レスポンシブ対応・構築支援を担当。
発注元企業様の制作体制に合わせ、クライアント名・URL・画面キャプチャは掲載していません。
```

NG：

```txt
制作実績：株式会社〇〇様
制作実績：沖縄料理〇〇様
URLはこちら
画面キャプチャはこちら
DF CONNECTが制作しました
```

OK：

```txt
匿名対応実績：既存サイトのCMS移行・レスポンシブ対応
匿名対応実績：飲食店サイトのHTML再構築・CMS設定
匿名対応実績：企業サイトの移設・配色調整・マルチデバイス対応
```

---

## 1-3. デモサイトは「品質確認用デモ」として前面に出す

今回添付されているスクリーンショットは、LPに載せる価値が高い。  
ただし、扱い方が重要。

スクリーンショット：

```txt
haru.png    ：飲食業向けデモ / カフェ・飲食店系サイト
livesta.png ：不動産業向けデモ / 不動産・開発会社系サイト
tpi.jpeg    ：製造業向けデモ / 工場・製造業系サイト
```

LPではこれらを「実績」と断定せず、以下のように扱う。

```txt
品質確認用デモサイト
業種別デモサイト
制作品質確認用サンプル
デザイン・レスポンシブ・導線設計を確認できるデモ
```

仮に本当に実績として出せる許諾がある場合でも、今回のLPでは「デモサイト」と「匿名対応実績」を分けた方が強い。

理由：

```txt
・デモは見た目の品質確認に使える
・匿名実績は実案件での対応範囲確認に使える
・ホワイトラベル案件の守秘姿勢を崩さずに済む
・“実績を出せない弱さ”ではなく“守秘を重視している強さ”に変換できる
```

# 2. 現在LPの課題

既存LPは方向性としては悪くないが、まだ「汎用的なBtoBサービスLP」っぽい。  
Web制作会社向けホワイトラベル下請けLPとして見ると、以下が弱い。

## 2-1. 「証拠」の出し方が弱い

現在は「導入イメージ」セクションがあるが、実際に見込み客が確認したいものは以下。

```txt
・どの程度の見た目のサイトを作れるのか
・スマホ対応までちゃんとしているのか
・業種別に対応できそうか
・実案件で何を担当してきたのか
・なぜ実績を公開していないのか
・NDAやホワイトラベル案件の扱い方を理解しているのか
```

今の「Case 01 / Case 02 / Case 03」だけでは、少し抽象的。  
ここを「デモサイト」と「匿名対応実績」に分割する。

---

## 2-2. 「DF CONNECTがすべて解決します」がやや大げさ

この表現は、エンド顧客向けLPならまだ許容できるが、Web制作会社向けだと少し雑に見える。

Web制作会社はプロなので、こういう大げさな表現より、以下のような現実的な表現の方が刺さる。

```txt
制作会社の裏側に必要な実務を、守秘前提で支えます
案件が増えた時に、品質と納期を崩さない外部制作体制を
公開できない案件が多いからこそ、情報管理と再現性を重視しています
```

修正方針：

```txt
Before：DF CONNECTがすべて解決します
After ：制作会社の裏側に必要な実務を、守秘前提で支えます
```

---

## 2-3. CTAが「資料ダウンロード」に寄りすぎている

現在のCTA：

```txt
お問い合わせ・ご相談はこちら
サービス資料をダウンロード
```

悪くはないが、今回のLPで一番見せたいのは「デモサイト」と「匿名対応実績」。  
資料請求より、まず品質確認に誘導する方が自然。

修正後CTA：

```txt
デモサイトを見る
匿名実績について相談する
```

または：

```txt
デモサイトを確認する
まずは外注可否を相談する
```

おすすめ：

```txt
Primary CTA：デモサイトを見る
Secondary CTA：匿名実績・対応範囲を相談する
```

# 3. LP全体の推奨構成

既存LPを以下の順番に組み替える。

```txt
1. Header
2. Hero
3. Problems
4. White-label / NDA policy
5. Reasons / Solution
6. Services
7. Demo sites
8. Anonymous cases
9. Coverage / Tech stack
10. Security / workflow
11. Pricing or Engagement types
12. FAQ
13. Final CTA
14. Footer
```

現在のLPには `#cases` があるが、これをそのまま使うのではなく、以下に分割する。

```txt
#demo-sites
#anonymous-cases
```

ナビゲーションには `デモ・実績` として出す。

# 6. Problems修正指示

現在の「こんなお悩みはありませんか？」は活かせる。  
ただし、もう少し制作会社のリアルな課題に寄せる。

## 6-1. 推奨カード文言

```txt
1. 案件は増えているが、手を動かせる制作リソースが足りない
2. 外注先に任せたいが、クライアント情報や品質管理が不安
3. デザイン支給後の実装やWordPress化だけを任せたい
4. 公開後の軽微修正・保守対応が社内に滞留している
5. 繁忙期だけ柔軟に手伝ってくれる外部パートナーがほしい
```

## 6-2. トーン

「あなたの悩みを全部解決」ではなく、制作会社の現場感に寄せる。

見出し案：

```txt
制作会社の現場で、こんな詰まりはありませんか？
```

サブコピー：

```txt
新規案件だけでなく、下層ページ制作、既存サイトの改修、公開後の更新作業まで、社内リソースを圧迫する細かな実務を外部化できます。
```

# 7. 新設：White-label / NDA policy セクション

現在のLPはNDA対応を打ち出しているが、「実績を公開しないことの意味」を説明するセクションが足りない。  
ここを入れると、実績を見せにくい弱点が強みに変わる。

## 7-1. セクション位置

Problemsの直後、Reasonsの前に入れる。  
理由：見込み客が「実績は？」と感じる前に、先に守秘方針を提示するため。

## 7-2. 見出し

```txt
公開実績より、守秘と再現性を重視しています
```

## 7-3. 本文

```txt
DF CONNECTでは、Web制作会社様・広告代理店様の裏側で動くホワイトラベル案件を中心に対応しています。
そのため、クライアント名・URL・画面キャプチャを大きく掲載することよりも、発注元企業様のブランドと案件情報を守ることを優先しています。

実案件の詳細は匿名化した対応実績としてご紹介し、制作品質については業種別のデモサイトでご確認いただけるようにしています。
```

## 7-4. 3カード

```txt
カード1：クライアント非接触
本文：原則としてエンドクライアント様への直接連絡は行わず、貴社の制作体制に合わせて裏側で対応します。

カード2：NDA・情報管理
本文：案件開始前のNDA締結、共有ファイル・アカウント権限・連絡ルールの整理に対応します。

カード3：実績は匿名化
本文：公開許諾のない案件は、社名・URL・画面キャプチャを掲載せず、担当範囲のみ匿名でご紹介します。
```

# 8. Reasons修正指示

現在：

```txt
DF CONNECTがすべて解決します
```

修正後：

```txt
制作会社の裏側に必要な実務を、守秘前提で支えます
```

## 8-1. 5つの理由

現在の5カードは方向性として使える。  
ただし文言を制作会社向けに微修正する。

```txt
1. ホワイトラベル対応
貴社ブランドでの納品を前提に、クライアント対応・表記ルール・連絡経路に配慮して制作します。

2. NDA・機密保持対応
案件開始前のNDA締結、共有範囲の整理、アカウント権限管理など、外注時の不安を減らします。

3. 制作〜運用保守まで対応
新規制作だけでなく、公開後の更新、軽微修正、WordPress保守、バックアップ確認まで継続対応できます。

4. スポット・継続どちらも対応
繁忙期の単発支援から、月額保守の外部パートナーまで、必要な範囲だけ柔軟に参加します。

5. 実務寄りの対応力
派手な提案だけでなく、下層ページ制作、既存サイト改修、フォーム調整、公開前チェックなど細かな作業を拾えます。
```

# 9. Services修正指示

現在のサービス内容は広すぎる。  
Web制作会社向けに「発注しやすい業務単位」で整理する。

## 9-1. 推奨サービスカード

現在は以下：

```txt
Webサイト制作
ECサイト構築
運用・保守サポート
コンテンツ・集客支援
各種連携・開発
```

修正後：

```txt
1. デザイン支給コーディング
2. WordPress制作・改修
3. LP制作・下層ページ制作
4. 運用保守・軽微修正代行
5. 公開前チェック・改善補助
```

理由：

```txt
・制作会社が外注しやすい粒度になる
・デザイン力の実績不足を補いやすい
・“手を動かす裏方”としての価値が伝わる
・運用保守の継続収益につながりやすい
```

## 9-2. サービスカード文言

### 1. デザイン支給コーディング

```txt
Figma・XD・画像カンプなどのデザインデータをもとに、HTML/CSS/JavaScriptで実装します。レスポンシブ対応、アニメーション、公開前チェックまで対応可能です。

対応例：
・トップページ実装
・下層ページ実装
・LPコーディング
・レスポンシブ対応
・軽微なJS実装
```

### 2. WordPress制作・改修

```txt
WordPressサイトの新規構築、既存テーマの改修、固定ページ追加、投稿タイプ調整、フォーム設置などに対応します。

対応例：
・WordPress化
・既存テーマ修正
・固定ページ追加
・投稿 / カスタム投稿調整
・Contact Form 7等のフォーム調整
```

### 3. LP制作・下層ページ制作

```txt
制作会社様の案件増加時に、LPや下層ページの制作リソースとして参加します。既存トンマナを踏襲したページ追加にも対応します。

対応例：
・サービスLP制作
・採用LP制作
・下層ページ量産
・既存サイトへのページ追加
・CTA / フォーム導線調整
```

### 4. 運用保守・軽微修正代行

```txt
公開後に発生するテキスト修正、画像差し替え、WordPress更新、バックアップ確認、フォーム動作確認などを代行します。

対応例：
・月次更新作業
・軽微なレイアウト調整
・画像差し替え
・バックアップ確認
・プラグイン更新補助
```

デモカードでは画像を大きく使う。

推奨CSS：

```css
.demo-card__image {
  display: block;
  overflow: hidden;
  border-radius: 18px 18px 0 0;
  background: #05070d;
  border-bottom: 1px solid rgba(216, 227, 239, 0.7);
}

.demo-card__image img {
  width: 100%;
  aspect-ratio: 16 / 8.2;
  object-fit: contain;
  background: #05070d;
  transition: transform 0.28s ease, opacity 0.28s ease;
}

.demo-card:hover .demo-card__image img {
  transform: scale(1.025);
  opacity: 0.96;
}
```

`tpi.jpeg` は白背景が含まれるため、`object-fit: contain` を推奨。

## 20-1. デモURL未設定時の処理

本番公開までにデモURLを入れるならJS不要。  
もし仮で `href="#"` のまま置く場合は、クリックできないようにする。

ただし、公開LPで `準備中` を見せるのは弱い。  
できればデモURLを必ず用意する。

暫定JS：

```js
const disabledDemoLinks = document.querySelectorAll('[data-demo-disabled]');
disabledDemoLinks.forEach((link) => {
  link.addEventListener('click', (event) => {
    event.preventDefault();
    alert('デモサイトは現在準備中です。お問い合わせ時に個別にご案内します。');
  });
});
```

ただし、これは最終手段。  
LPの説得力を考えると、デモサイトURLを先に用意する方がよい。

```txt
10年以上前に制作された企業サイトを、既存の印象を大きく崩さずにCMS運用へ移行。配色調整、レスポンシブ対応、公開前確認までを支援しました。

Case 02
飲食店サイトのHTML再構築・CMS対応
既存デザインを踏襲しながらHTMLを再構築し、スマホ・タブレット表示に対応。店舗側で情報更新しやすいCMS項目の設定も支援しました。

Case 03
制作会社の下請け制作・改修支援
Web制作会社様からのデザイン支給後、下層ページの実装やレスポンシブ調整を担当。連絡経路・表記ルール・納品形式を発注元企業様に合わせて対応しました。

Case 04
公開後の運用保守・軽微修正代行
公開後に発生する細かな更新作業や保守対応を代行。制作会社様が新規案件やディレクションに集中できるよう、継続的な運用タスクを支援します。
```

# 23. NG表現リスト

以下はLPに入れない。

```txt
制作実績多数
大手実績あり
何でもできます
業界最安
最短即日で高品質
完全丸投げOK
DF CONNECTがすべて解決します

クライアント名を無許可で掲載
M's Create様経由案件をDF CONNECT単独制作のように見せる
デモサイトを実案件のように見せる
```

理由：

```txt
・Web制作会社向けには誇張表現より実務信頼が重要
・ホワイトラベル案件で実績公開を強く出すと逆に不信感になる
・実績の見た目が弱い場合、スクリーンショット公開は逆効果
・協力案件の表記を誤ると信用を失う
```

# 25. デモサイトと実績の表記ルール

## 25-1. デモサイト

表記：

```txt
品質確認用デモサイト
業種別デモサイト
デモサイト
```

目的：

```txt
見た目、レスポンシブ、導線設計、構成力を確認してもらう
```

使ってよい画像：

```txt
haru.png
livesta.png
tpi.jpeg
```

ただし、「実際に納品したサイト」とは書かない。

---

## 25-2. 匿名対応実績

表記：

```txt
匿名対応実績
対応実績
実案件の対応内容
```

目的：

```txt
実案件で何を担当してきたかを伝える
```

見せる内容：

```txt
・対応内容
・担当範囲
・対応フェーズ
・公開範囲
```

見せない内容：

```txt
・クライアント名
・URL
・スクリーンショット
・発注元名
・担当者名
```

# 26. LPの訴求ストーリー

このLPのストーリーは以下にする。

```txt
1. Web制作会社は案件増加・保守対応・細かい修正で詰まる
2. でも外注には、情報漏洩・品質・納期・クライアント接触の不安がある
3. DF CONNECTは、NDA対応・ホワイトラベル・クライアント非接触で裏側支援できる
4. 実績は守秘のため大きく公開しない
5. その代わり、デモサイトで品質を確認できる
6. 実案件の経験値は匿名対応実績として確認できる
7. スポット制作から保守まで、必要な範囲で相談できる
```

LP全体でこの流れが崩れないようにする。

# 30. 最終的なLPの見え方

ユーザーがLPを見た時、こう感じるのが理想。

```txt
・派手な制作会社ではなく、裏側でちゃんと動いてくれそう
・実績を勝手に公開しなさそう
・NDAやホワイトラベル案件の扱いを分かっていそう
・実績は匿名だけど、デモサイトで品質は確認できる
・小さい修正や保守も頼みやすそう
・デザイン支給後の実装を任せやすそう
・制作会社側の都合に合わせて動いてくれそう
```

逆に、こう見えたら失敗。

```txt
・普通のホームページ制作会社に見える
・実績が弱く見える
・デモと実績の違いが曖昧
・何でも屋っぽい
・守秘対応を本当に理解しているか不安
・問い合わせる前に品質確認できない
```

```txt
実績を大きく見せることより、

```

これが今回のLPの核。

# 34. 最後に

今回のLPは「制作会社として前に出るLP」ではなく、「制作会社の裏側で信用されるLP」にする。  
だから、実績を無理に出すより、出さない理由を丁寧に説明した方が強い。

最終的な一文はこれ。

```txt
表に出ないからこそ、安心して任せられる。
Web制作会社のためのホワイトラベル制作・運用保守パートナー。
```

この世界観から外れないように、コピー、画像、CTA、FAQ、実績表記を全て統一する。
