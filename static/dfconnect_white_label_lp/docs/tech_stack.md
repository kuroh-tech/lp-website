## 2. 受注範囲拡大で導入を検討すべき技術スタック

現状の「コーディング＋WordPress」を軸に、段階的に対応領域を広げる場合のロードマップ。
優先度は「制作会社からの受注で実際に求められる頻度」で判定。

### Phase 1 — 既存案件の対応幅を広げる（最優先）

制作会社の既存プロジェクトに参画する際に「これができないと入れない」になりやすいもの。

#### Sass / SCSS

| 項目 | 内容 |
|------|------|
| なぜ必要か | 大手・中堅制作会社のプロジェクトはSass前提が多い。既存プロジェクトのCSSがSassで書かれている場合、Plain CSSしか書けないと参画できない |
| 学習コスト | 低い。現状のCSSカスタムプロパティ設計ができていれば、変数・ネスト・ミックスイン・パーシャル分割を覚えるだけ |
| 導入タイミング | 次の新規案件から。既存案件を書き換える必要はない |
| 備考 | SCSS（Sassのインデント記法ではなくブレース記法）が主流 |

#### CSS設計手法（BEM / FLOCSS）

| 項目 | 内容 |
|------|------|
| なぜ必要か | 制作会社のコーディングガイドラインに「BEM」「FLOCSS」が指定されているケースが多い。命名規則を合わせないと納品NGになる |
| 学習コスト | 低〜中。概念自体はシンプルだが、既存プロジェクトの設計に合わせる応用力が必要 |
| 導入タイミング | 指定があった案件から。自分のプロジェクトでBEMを標準にするかは任意 |

#### ビルドツール（Vite）

| 項目 | 内容 |
|------|------|
| なぜ必要か | Sassのコンパイル、JS/CSSの結合・圧縮、画像最適化、ホットリロード。制作会社のプロジェクトで「npm run dev で立ち上がる」構成が増えている |
| 学習コスト | 中。npm/Node.jsの基礎知識が前提。Vite自体の設定は軽量 |
| 導入タイミング | Sass導入と同時。Sass単体よりVite経由で使う方が実務的 |
| 備考 | webpack は設定が重いので新規で選ぶ必要はない。Viteで十分。既存プロジェクトがwebpackの場合は読めればOK |

#### EJS / Pug（HTMLテンプレートエンジン）

| 項目 | 内容 |
|------|------|
| なぜ必要か | 静的サイトのコーディングで、ヘッダー・フッターの共通パーツ管理に使われる。Viteと組み合わせて使うケースが多い |
| 学習コスト | 低い。PHPのinclude/requireと同じ概念 |
| 導入タイミング | Vite導入後 |

### Phase 2 — 単価を上げる・案件の幅を広げる

コーディング単体ではなく「実装＋α」として請けられるようになるもの。

#### JavaScript（ES6+の深掘り）

| 項目 | 内容 |
|------|------|
| なぜ必要か | 現状のVanilla JSはUI制御中心。非同期処理（fetch/async-await）、モジュール分割（import/export）、API連携ができると、フォームの非同期送信、外部データ連携、SPAライクな動きに対応できる |
| 学習コスト | 中。基礎はできているので、モジュール設計と非同期処理の実践 |
| 効果 | 「JSも書ける」ではなく「JSで実装設計ができる」になる。単価が変わる |

#### PHP（フレームワーク：Laravel）

| 項目 | 内容 |
|------|------|
| なぜ必要か | WordPress以外のPHP案件（管理画面、フォーム高機能化、API構築）を請ける場合に必要。現状のクラス設計力があればLaravelの理解は早い |
| 学習コスト | 中〜高。フレームワークの規約、Eloquent ORM、Blade、認証、ミドルウェア |
| 導入タイミング | WordPressでは対応しきれない案件が来た時。先回りで学ぶよりは案件駆動でOK |
| 備考 | 「大規模なシステム開発」は対応外としているが、Laravelを使った中小規模の管理画面・業務ツールは受注範囲に入りうる |

#### TypeScript

| 項目 | 内容 |
|------|------|
| なぜ必要か | JSの品質を上げる。フロントエンド開発案件では型定義が標準化しつつある。React/Vue案件に入る場合の前提知識 |
| 学習コスト | 中。JS自体はわかっているので、型システムの概念と書き方 |
| 導入タイミング | Phase 3のフレームワーク導入前に |

#### WordPress（Advanced Custom Fields / カスタムブロック）

| 項目 | 内容 |
|------|------|
| なぜ必要か | 制作会社のWP案件で「ACFで投稿画面をカスタマイズ」は頻出。カスタムブロック開発（React + PHP）ができるとWP案件の単価が上がる |
| 学習コスト | ACF：低い。カスタムブロック：中〜高（React知識が前提） |
| 導入タイミング | ACFは即導入可能。カスタムブロックはPhase 3以降 |

### Phase 3 — 開発案件を請ける

「コーディングパートナー」から「フロントエンド開発パートナー」へのシフト。

#### React

| 項目 | 内容 |
|------|------|
| なぜ必要か | フロントエンド開発の主流。制作会社が開発案件を外注する場合、Reactが最も多い。WordPressのカスタムブロック開発にもReact（JSX）が必要 |
| 学習コスト | 高い。コンポーネント設計、状態管理、フック、レンダリング最適化 |
| 導入タイミング | TypeScript習得後 |

#### Next.js

| 項目 | 内容 |
|------|------|
| なぜ必要か | Reactベースのフレームワーク。SSR/SSG/ISRによるパフォーマンス最適化。ヘッドレスCMS連携。制作会社がWP以外のCMS案件で採用するケースが増加 |
| 学習コスト | 高い。React + ルーティング + データ取得 + デプロイ |
| 導入タイミング | React習得後。案件が見えてから |

#### ヘッドレスCMS（microCMS / Newt / Strapi）

| 項目 | 内容 |
|------|------|
| なぜ必要か | WordPress以外のCMS選択肢。Next.js/Astroとの組み合わせで使われる。制作会社が「WPじゃないCMSで」と言い始めた時に対応できる |
| 学習コスト | 低〜中。API経由のデータ取得が中心なのでフロント側の知識で対応可能 |
| 導入タイミング | Next.js or Astro導入後 |

#### Astro

| 項目 | 内容 |
|------|------|
| なぜ必要か | 静的サイト生成に特化。コンテンツ中心のサイト（コーポレート、LP、ブログ）ではNext.jsより軽量で実用的。制作会社が「WordPressは重い、でもCMSは欲しい」と言った時の回答 |
| 学習コスト | 中。HTML/CSS/JSの延長で入れる。React/Vueコンポーネントも混在可能 |
| 導入タイミング | Next.jsと並行 or 先に。静的サイトが主戦場なら先に学ぶ方が実務に近い |

### Phase 4 — 長期的に視野に入れておくもの

今すぐ不要だが、業界の流れとして知っておくべきもの。

| 技術 | 理由 | 緊急度 |
|------|------|--------|
| Vue.js / Nuxt | React以外の選択肢。制作会社によってはVue派がいる | 低い。Reactを先にやる |
| Tailwind CSS | ユーティリティファーストCSS。開発速度重視のプロジェクトで増加中 | 低い。案件で指定があったら対応 |
| Storybook | UIコンポーネントのカタログ管理。大規模プロジェクト向け | 低い。チーム開発時に必要になったら |
| CI/CD（GitHub Actions） | テスト自動化、自動デプロイ。開発案件で求められる | 低い。Phase 3以降で自然に触れる |
| Figma API / デザイントークン | デザインとコードの連携自動化。大手案件で増加中 | 低い。知識として持っておく程度 |

## 3. 導入優先度のまとめ

```
今すぐ（Phase 1）
├── Sass / SCSS
├── BEM / FLOCSS（案件指定時）
├── Vite（Sassと同時）
└── EJS / Pug（Viteと同時）

半年以内（Phase 2）
├── JavaScript ES6+ 深掘り（async/await, modules）
├── TypeScript
├── ACF（WordPress）
└── Laravel（案件が来たら）

1年以内（Phase 3）
├── React
├── Next.js or Astro
├── ヘッドレスCMS
└── WordPress カスタムブロック

把握しておく（Phase 4）
├── Vue.js / Nuxt
├── Tailwind CSS
├── Storybook
├── CI/CD
└── Figma API
```
