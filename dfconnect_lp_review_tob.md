# DF CONNECT LP — ToB向けレビュー

制作会社のディレクター・PMが見た時に「この人に頼めるか判断できるか」を軸に洗い出し。

---

## A. 足りていないもの

### A-1. コーディングの技術的な具体性

現状、「コーディング・LP実装」カードの中身が「Figma / XD / 画像カンプからのHTML/CSS実装、レスポンシブ対応」の1文だけ。
制作会社はここが一番知りたい。「コーディングできます」は誰でも言える。

**追加すべき情報：**

- 対応できるマークアップ：セマンティックHTML、BEM or FLOCSS等の命名規則対応
- CSS：Sass/SCSS対応、CSS設計の方針（カスタムプロパティ、メディアクエリの書き方）
- JavaScript：Vanilla JS、jQuery、軽量なUIライブラリ（Swiper等）の組み込み
- レスポンシブ：モバイルファースト or デスクトップファースト、ブレイクポイントの擦り合わせ
- アクセシビリティ：WAI-ARIA、キーボード操作、alt属性、フォーカス管理
- パフォーマンス意識：画像最適化、Core Web Vitals意識

→ LPの「対応内容」セクション内、もしくはコーディング専用の小セクションとして見せる。
箇条書きで並べるだけで制作会社の安心材料になる。

### A-2. 納品形式・作業環境

制作会社が外注先を選ぶとき、「この人はうちのワークフローに合うか」を見る。

**追加すべき情報：**

- 納品形式：HTML/CSSファイル一式、WordPress直接構築、Git経由、FTP、zip納品
- バージョン管理：Git対応（GitHub / GitLab / Backlog Git）
- 連絡ツール：Slack、Chatwork、Backlog、メール等（対応可能なもの列挙）
- デザイン受け取り：Figma、XD、Photoshop、画像カンプ（PDF / PNG）
- 開発環境：ローカル環境構築、テストサーバー確認

→ 進め方セクションに入れるか、対応内容セクションの下に「作業環境・納品形式」として短く1ブロック追加。

### A-3. 目安の納期感

「どのくらいで上がるか」はPMが最初に考えること。今のLPにはゼロ。

**追加例（断言でなく目安）：**

- LP実装（1ページ）：5〜10営業日目安
- 下層ページ追加：2〜5営業日目安
- WordPress固定ページ追加：2〜3営業日目安
- 軽微修正：1〜2営業日目安

→ 「案件規模により変動します」の注記付きで、対応内容カードまたはFAQに入れる。
目安がないと、そもそもスケジュールに組めるかの判断すらできない。

### A-4. 料金の目安

仕様書で「控えめに」とあったので現状ゼロだが、制作会社は相見積もりが日常。
目安がないと「問い合わせるまで全くわからない」＝ 問い合わせのハードルになる。

**追加案（攻めすぎない範囲）：**

- LP実装：¥50,000〜（デザイン支給、1ページ構成の場合）
- WordPress固定ページ追加：¥15,000〜
- 軽微修正：¥5,000〜 / 件

→ セクションとして独立させず、対応内容カードの下に小さく「参考価格帯」として添える程度。
または FAQ に「目安を教えてください」として入れる。
制作会社は「自社の外注費枠に合うか」を知りたいだけ。

### A-5. 品質保証プロセス

制作会社が外注先に不安を感じるのは「上がったものの品質」。

**追加すべき情報：**

- クロスブラウザ確認：Chrome、Safari、Edge、Firefox（対応範囲の明示）
- デバイス確認：実機確認の有無、BrowserStack等
- バリデーション：HTMLバリデーション、リンクチェック
- 表示確認のタイミング：初稿確認→修正→最終確認の流れ

→ 進め方セクションのStep 5〜6に組み込むか、品質確認に触れる小見出しを1つ追加。

## B. 強めた方がいいもの

### B-1. コーディングを主役に据える

現状、4つの対応内容（コーディング、WordPress、運用保守、公開前チェック）が均等に並んでいる。
ユーザー自身が「まずはコーディングから」と言っている以上、**コーディングがメインスキルであることを明確にすべき**。

**具体策：**

- 対応内容セクションで、コーディングカードを他より目立たせる（サイズ、色、上部配置）
- FVリード文でも「デザイン支給後のコーディング」が最初に来ているのは良い。ただしもう少し踏み込める
- 例：「Figma支給からのセマンティックHTML/CSS実装を中心に、制作会社様の現場で必要な実装作業を対応します」

### B-2. デモサイトの技術的な見せ方

デモサイト3件は良い資産。しかし今は「想定業種」と「確認できるポイント」が見た目寄りの説明。
**制作会社が見るのはコードの質**。

**強化案：**

- 各デモカードに技術スペックを追加：
  - 「セマンティックHTML / Sass / レスポンシブ / Lighthouse Performance 90+」のようなタグ
  - 「ソースコードもご確認ください」の一言
- デモサイト自体のLighthouseスコアが良ければ、それを小さく記載する
- 制作会社のディレクターは「デモを見る→DevToolsでソースを確認」が普通。そこに誘導する姿勢が信頼になる

### B-3. 「制作会社の現場で詰まりやすい作業」セクション

現状の3枚カードは抽象的。制作会社のPMが「あ、うちのことだ」と思える具体性がもっとほしい。

**強化例：**

- 「案件はあるのに、制作リソースが追いつかない」
  → 「デザインは上がったのに、コーディングに着手できず公開が延びている」
- 「公開前チェック、軽微修正が後回しになりやすい」
  → 「公開前のリンクチェック・フォーム確認・OGP設定が積み残しになっている」
- 「外部依頼したいが、進行ルールと情報開示の線引きを整理したい」
  → 「外注したいが、クライアント情報の共有範囲や進行ルールを擦り合わせる手間が気になる」

→ 制作会社の日常言語に寄せるだけで刺さり方が変わる。

### B-4. 匿名実績の担当作業をもっと具体的に

現状の「設計・構築の一部、レスポンシブ対応、CMS対応」は抽象的すぎる。
守秘義務内でも、もう少し制作会社が技術レベルを推測できる情報を出せる。

**強化例：**

- 「Figma支給 → HTML/CSS/JSコーディング → WordPress化（カスタム投稿タイプ、固定ページテンプレート）→ レスポンシブ対応 → 表示確認・修正対応」
- 「既存HTML構造の再設計 → Sass導入 → CMS組み込み → クロスブラウザ確認」
- ページ数やおおよその工期を添えると規模感が伝わる（例：「約15ページ、実装期間3週間」）

### B-5. FVの「個人事業主が直接対応」チップ

これは現状、やや防御的に聞こえる。
制作会社がほしいのは「小回りが利く」「レスポンスが早い」「伝言ゲームにならない」。

**言い換え案：**

- 「個人事業主が直接対応」→「担当者=制作者、伝言ゲームなし」
- 「相談窓口と実装者が同一人物」

→ 個人事業であることの"結果としてのメリット"で訴求する。

## C. 優先度まとめ

| 優先度 | 項目 | 理由 |
|--------|------|------|
| **高** | A-1 コーディングの技術的な具体性 | 主力サービスの判断材料がない |
| **高** | B-1 コーディングを主役に据える | 何屋かが一目で分からない |
| **高** | A-2 納品形式・作業環境 | 自社フローに合うか判断できない |
| **高** | B-2 デモの技術的な見せ方強化 | せっかくの資産が活きていない |
| **中** | A-3 納期感の目安 | スケジュールに組めるか判断できない |
| **中** | B-3 課題セクションの具体化 | 共感精度を上げる |
| **中** | B-4 匿名実績の具体化 | 技術レベルの推測材料 |
| **低** | A-4 料金の目安 | あった方がいいが、なくても問い合わせは来る |
| **低** | A-5 品質保証プロセス | 進め方に組み込めばOK |
| **低** | B-5 チップの文言微調整 | 小さい改善だが効く |

## D. 全体所感

現状のLPは「情報商材感を消す」「匿名実績を適切に見せる」「NDA前の相談ハードルを下げる」は十分できている。
フォーム設計、プライバシーポリシーの扱い、フッターの簡素化も良い。

**一方で、制作会社の技術担当やPMが「この人のコーディング力はどのレベルか」を判断する材料が薄い。**
LPの見た目やトーンは整っているが、**中身が「できます」の羅列で、「どうやっているか」「何を使っているか」が見えない**。

コーディング外注先を探している制作会社は、デモサイトのソースコードを見ればある程度判断できる。
が、**ソースを見る前にLPで落とされる可能性がある**。LPの文言で技術レベルのシグナルを出すことが先。

「開発もできるが、まずはコーディングから」であれば、コーディング特化の見せ方にした方が、
制作会社側の「この人は何が得意か」が明確になり、問い合わせ精度が上がる。

## E. 実装コードから裏付けた技術スタック

wordpressディレクトリのテーマ3件（haru / livesta / tpi）と、DF CONNECT LP本体のコードを精査した結果。
**実際に使っている技術のみ記載。盛らない。**

### E-1. マークアップ・CSS

| 項目 | 実装状況 |
|------|----------|
| セマンティックHTML | `<section>`, `<article>`, `<nav>`, `<main>`, `<details>` 等を標準的に使用 |
| CSS設計 | CSSカスタムプロパティ（CSS変数）ベースの設計。テーマごとに色・影・余白を変数化 |
| CSS命名規則 | BEMライクな命名（厳密なBEMではなく、コンポーネント単位の命名） |
| レスポンシブ | デスクトップファースト（WPテーマ3件）、モバイルファースト（DF CONNECT LP） |
| ブレイクポイント | 案件に応じて設定（920px / 640px、1080px / 860px、960px / 768px / 420px 等） |
| モダンCSS | `clamp()`, `min()`, CSS Grid, Flexbox, `scroll-behavior`, `clip-path` |
| レイアウト | CSS GridとFlexboxの使い分け。Grid=2〜3カラム構成、Flex=ヘッダー・ナビ・カード内部 |
| CSSプリプロセッサ | 未使用（Plain CSS）。Sass/SCSS対応は可能だが、現行はカスタムプロパティで設計 |
| Webフォント | Google Fonts（Noto Sans JP, Zen Kaku Gothic New） |

### E-2. JavaScript

| 項目 | 実装状況 |
|------|----------|
| ライブラリ依存 | なし。全テーマ・LP共にVanilla JS |
| 設計パターン | IIFE（即時実行関数）でスコープ管理 |
| 主な実装 | モバイルメニュー開閉、スクロール検知（sticky header）、IntersectionObserver（スクロールアニメーション）、アコーディオン |
| ARIA操作 | `aria-expanded`, `aria-controls`, `aria-label` をJSで動的に切り替え |
| キーボード対応 | Escキーでメニュー閉じ、リサイズ時の自動閉じ |
| jQuery | 未使用 |

### E-3. WordPress

| 項目 | 実装状況 |
|------|----------|
| テーマ開発 | フルスクラッチのオリジナルテーマ（既存テーマ・テーマビルダー不使用） |
| テンプレート構成 | header / footer / front-page / page / single / archive / 404 / index |
| カスタムページテンプレート | page-about / page-contact / page-service / page-privacy / page-sitemap / page-thanks（livesta） |
| カスタム投稿タイプ | `property` CPT + カスタムタクソノミー2種（livesta） |
| functions.php規模 | 660行（tpi）〜 1,378行（livesta）。データ構造を関数で管理 |
| テーマサポート | title-tag, post-thumbnails, HTML5（search-form, comment-form, gallery等） |
| ナビゲーション | `register_nav_menus()` でメニュー位置を登録 |
| プラグイン依存 | 最小（akismetのみ）。機能はテーマ内で完結 |
| PHP | 8.2（Docker環境） |

### E-4. バックエンド（DF CONNECT LP）

| 項目 | 実装状況 |
|------|----------|
| 言語 | PHP（Vanilla、フレームワーク不使用） |
| アーキテクチャ | クラスベース（Validator, Csrf, RateLimiter, InquiryRepository, MailService） |
| フォーム処理 | CSRF保護、レートリミット、バリデーション、セッションFlashメッセージ |
| メール | PHPMailer（Composer経由）によるSMTP送信。管理者通知+自動返信の2通構成 |
| DB | MySQL/MariaDB。inquiriesテーブルに問い合わせ保存 |
| セキュリティ | CSRF token、ハニーポット、レートリミット、入力サニタイズ |
| ログ | storage/logs/ にエラーログ出力 |

### E-5. 開発環境・インフラ

| 項目 | 実装状況 |
|------|----------|
| ローカル開発 | Docker Compose（MariaDB 11.4 + WordPress PHP8.2-Apache） |
| バージョン管理 | Git（GitHub） |
| サーバー | XServer想定（.htaccess、xserver-db-config.php.example が存在） |
| デプロイ | zip + FTPアップロード（archives/に配置用zipあり） |

### E-6. アクセシビリティ

| 項目 | 実装状況 |
|------|----------|
| スキップリンク | DF CONNECT LPで実装済み |
| WAI-ARIA | 全テーマ・LPで `aria-expanded`, `aria-controls`, `aria-label`, `aria-hidden`, `aria-required`, `aria-invalid`, `aria-describedby` を使用 |
| フォーム | エラー時の `role="alert"`, `aria-live="assertive"` で動的通知 |
| 動作軽減 | `prefers-reduced-motion` メディアクエリでアニメーション無効化（DF CONNECT LP） |
| sr-only | `clip-path: inset(50%)` による視覚非表示テキスト |

## F. LP掲載用の技術スタック案（実装ベース、盛らない）

LPに載せる場合の文言案。実際にコードで確認できたものだけ。

### コーディング

```
HTML：セマンティックHTML（section / article / nav / details 等）
CSS：CSSカスタムプロパティ設計、CSS Grid / Flexbox、clamp() / min() によるレスポンシブ、メディアクエリ設計
JS：Vanilla JavaScript（IntersectionObserver、ARIA制御、モバイルメニュー、スクロール制御）
レスポンシブ：モバイルファースト / デスクトップファースト、案件に応じてブレイクポイント設計
アクセシビリティ：WAI-ARIA、キーボード操作、prefers-reduced-motion対応、フォームのエラー通知
```

### WordPress

```
オリジナルテーマ開発（フルスクラッチ）
固定ページテンプレート、カスタム投稿タイプ、カスタムタクソノミー
functions.phpでのデータ管理、メニュー登録、アセット読み込み
プラグイン最小構成（テーマ内で機能完結）
```

### 対応ツール・納品形式

```
デザイン受け取り：Figma / XD / Photoshop / 画像カンプ（PDF / PNG）
バージョン管理：Git（GitHub）
納品形式：HTML/CSSファイル一式 / WordPress直接構築 / zip納品 / FTP
開発環境：Docker（ローカル環境構築対応）
サーバー：XServer等の共用サーバー対応
```

### 現時点でLPに書かない方がいいもの

- Sass/SCSS → 現行コードでは未使用。「対応可能」とは書けるが実績ベースではない
- jQuery → 使っていない。あえて書かない方がモダンな印象
- React / Vue / Next.js → 現行コードに存在しない。将来対応するなら別途追加
- BEM / FLOCSS → 厳密には採用していない。「BEMライク」と書くのも微妙なので「コンポーネント単位の命名設計」程度
